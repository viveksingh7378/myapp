New: Lint Stage

Added between Checkout and Tests, the Lint stage installs flake8 and scans all Python files in app/ and tests/. Errors are logged to lint_output.txt but are non-blocking — the pipeline reports the count and moves on.

Jenkinsfile — Lint stage (NEW)
stage('Lint') {
    steps {
        sh 'pip3 install flake8 --quiet'
        sh 'flake8 app/ tests/ --max-line-length=100 2>&1 | tee lint_output.txt || true'
        script {
            def lintErrors = sh(
                script: 'grep -c "E[0-9]\\|W[0-9]" lint_output.txt || true',
                returnStdout: true
            ).trim().toInteger()
            if (lintErrors > 0) {
                echo "Lint warnings found — see lint_output.txt"
            }
        }
    }
}

How the New Endpoints Work in app.py

app/app.py — POST /items (NEW)
@app.route("/items", methods=["POST"])
def create_item():
    data = request.get_json()
    if not data or "name" not in data:
        return jsonify({"error": "name is required"}), 400
    new_id = max(i["id"] for i in items) + 1 if items else 1
    new_item = {"id": new_id, "name": data["name"]}
    items.append(new_item)
    return jsonify(new_item), 201

@app.route("/items/<int:item_id>", methods=["DELETE"])
def delete_item(item_id):
    global items
    item = next((i for i in items if i["id"] == item_id), None)
    if item is None:
        return jsonify({"error": "not found"}), 404
    items = [i for i in items if i["id"] != item_id]
    return jsonify({"deleted": item_id}), 200

04 / AI Remediation Agent

The agent in ai_agent/ bridges test failure logs with Gemini's code understanding. It is called by Jenkins when tests fail and operates fully autonomously.

1

Parse the Log — log_parser.py

Scans test_output.txt for the first FAILED, ERROR, or SyntaxError keyword, then extracts 60 lines of surrounding context.

2

Query Gemini — Structured JSON Prompt

The context is sent to Gemini with a strict prompt demanding a raw JSON response: root_cause, file_path, original_code, fixed_code, confidence.

3

Model Fallback Chain

Tries gemini-2.0-flash-lite → gemini-2.0-flash → gemini-2.5-flash automatically on HTTP 429 quota errors.

4

Apply the Fix — apply_fix()

Only if confidence ≥ 0.75, performs an exact string replacement of the broken line in the source file. Refuses to proceed if the exact line isn't found.

5

Commit & Push — git_commit_fix()

Commits as AI-Remediation-Bot with message AI-Fix: <root_cause>, injects GitHub token into remote URL, then pushes to main to retrigger Jenkins.

🔄 Retry Guard: A .ai_retry_count file caps autonomous attempts at 2. If both fail, the agent exits with code 1 and Jenkins marks the pipeline as failed for human review. The counter resets at the start of each successful pipeline run.

ai_agent/remediate.py — expected Gemini JSON contract
# Gemini must return ONLY this — no markdown, no fences
{
  "root_cause":    "missing colon after function definition",
  "file_path":     "app/app.py",
  "original_code": "def create_item()",
  "fixed_code":    "def create_item():",
  "confidence":    0.97
}

05 / Docker & Kubernetes

Every image is tagged with the short git commit SHA and pushed to vivek7378/myapp:<sha> on Docker Hub. Trivy scans the image for HIGH/CRITICAL CVEs before push, saving trivy-report.json as a Jenkins artifact.

🔒 Security-first: Images are scanned before push. The archived JSON report can be consumed by security dashboards for CVE trend analysis. Scan is non-blocking (--exit-code 0) to surface warnings without halting the build.

Jenkinsfile — K8s deploy
sh """
    sed 's|IMAGE_TAG|${GIT_COMMIT_SHORT}|g' k8s/deployment.yaml \
        > k8s/deployment-final.yaml
    kubectl apply -f k8s/deployment-final.yaml
    kubectl apply -f k8s/service.yaml
    kubectl rollout status deployment/myapp --timeout=60s
"""

How It Works — Step by Step

1

You make a mistake in any file

For example: you delete <html> from blog.html, or remove a : from a Python function, or add a mismatched brace in CSS.

2

You push to GitHub → Jenkins triggers

Any commit to main starts the pipeline. The first real stage is now Universal Validate.

3

validator.py scans every file

It walks the whole project, skipping venv/ and caches, and runs the right checker for each file type. All errors go into validation_output.txt.

4

Jenkins detects FAILED in the output

If validation_output.txt contains FAILED or ERROR, Jenkins calls the AI agent with that log file.

5

Gemini analyzes the error in any language

The updated prompt tells Gemini the error could be Python, HTML, JS, CSS, or JSON — and to return the exact broken text and the fix as a JSON object.

6

Fix applied, committed, pushed — pipeline reruns

The agent patches the exact file, commits as AI-Remediation-Bot, and pushes. Jenkins re-triggers automatically on the new commit.

Example — Deleting <html> from blog.html

validation_output.txt — what validator.py writes
VALIDATOR FOUND 1 ERROR(S)

ERROR: blog.html: Missing required tag '<html'

FAILED
Gemini response — what the AI agent receives back
{
  "root_cause":    "blog.html is missing the opening <html> tag",
  "file_path":     "blog.html",
  "language":      "html",
  "original_code": "<!DOCTYPE html>\n lang=\"en\">",
  "fixed_code":    "<!DOCTYPE html>\n<html lang=\"en\">",
  "confidence":    0.98
}
